일반적으로 관계형 데이터베이스(RDB)를 조작할 때는 SQL를 사용한다.   
  
흔히 개발자는 프로그래밍을 하는 것과 같이, SQL을 통해서 데이터에 직접 접근하고 조작한다고 생각하지만 사실은 그렇지 않다. 개발자가 데이터베이스에서 의식적으로 사용하는 것은 SQL 레벨까지다. 이후의 모든 것은 SQL 구문을 읽어 들인 DBMS가 알아서 처리하고 결과를 제공해 준다. 따라서 개발자는 데이터가 있는 곳을 알 필요 없고, 데이터에 접근하는 방법도 따로 생각하기 않아도 된다. 결국 모든 것은 DBMS에게 맡겨지기 때문이다.

#### 데이터에 접근하는 방법은 어떻게 결정될까?

RDB에서 데이터에 접근 절차를 결정하는 모듈은 쿼리 평가 엔진이라고 부른다. 쿼리 평가 엔진은 사용자로부터 입력받은 SQL 구문을 처음 읽어 들이는 모듈이기도 하다. MySQL은 MySQL 엔진에서 동일한 기능을 수행한다. 쿼리 평가 엔진은 파서, 옵티마이저와 같은 여러 개의 서브 모듈로 구성된다.   
  
다음의 그림은 쿼리가 들어오는 시점 부터, 실제로 데이터 접근이 이루어지는 순서에 대한 개략적인 그림이다.  
  

![](/images/posts/dbms-39/screenshot-2023-10-04-at-9-53-12-pm.png)

-   **파서**: 구문 분석을 수행한다. 사용자로 부터 입력받은 SQL의 구문이 올바르게 작성되어 있는지 평가하고, 정형적인 형식으로 변환해 준다.
-   **옵티마이저**: 최적화를 수행하는 요소로 최적화의 대상은 데이터 접근법(실행 계획)이다. 옵티마이저는 DBMS의 두뇌의 역할을 수행한다. 옵티마이저는 인덱스의 유무, 데이터 분산 또는 편향 정도, DBMS 내부 매개변수 등의 조건을 고려해서, 선택 가능한 많은 실행 계획을 작성하는데, 이를 플랜 생성 단계에서 수행한다. 플랜 생성 단계에서 생성된 많은 실행계획 중 비용 평가 단계에서 수행 비용을 계산하고 최소 비용을 가진 실행 계획을 선별한다.
-   **카탈로그 매니저**: 옵티마이저가 실행 계획을 세울 때 옵티마이저에게 중요한 정보를 제공한다. 카탈로그에는 DBMS의 내부 정보를 모아놓은 테이블들이 있는데, 각 테이블에는 데이터 테이블 또는 인덱스의 통계 정보가 저장되어 있다. 이러한 정보가 저장되어 있어서 카탈로그를 통계 정보로 부르기도 한다.
-   **플랜 평가**: 옵타마이저가 도출한 최적의 실행 계획을 선택하는 것이 플랜 평가다. 옵티마이저에서 선택된 실행 계획은 DBMS에서 바로 실행할 수 있는 코드는 아니다. 그렇기 때문에 실행 계획을 절차적인 코드로 변환한 뒤에 데이터 접근을 수행해야 한다.

각 역할 중 통계 정보와 옵티마이저는 아주 의존적인 관계를 가지고 있다. 즉, 올바른 통계 정보는 SQL 성능에 있어서 굉장히 중요하다는 것을 의미한다. 만약 테이블 또는 인덱스의 상태와 통계 정보가 불일치하다면, 옵티마이저는 잘못된 정보를 기반으로 실행 계획을 세우기 때문에 문제가 발생할 수 있다. 그렇기 때문에 테이블의 데이터가 지속해서 변경되면, 카탈로그의 통계 정보도 바꾸어 주어야 한다. 하지만 통계 작업을 수행하는 것은 매우 비용이 큰 작업이기 때문에 적절한 갱신 시점을 검토한 뒤 작업을 수행해야 한다.   
  
실행 계획이 만들어지면 DBMS는 수립된 계획을 바탕으로 데이터에 접근하게 된다. 하지만 데이터 양이 많은 테이블에 접근하거나 복잡한 SQL 구문을 실행하면 최적의 실행 계획이더라도 반응 지연이 발생할 수 있다.   
  

_그렇다면 반응 지연이 발생했을 때 어떻게 대처해야 할까?_

  
가장 먼저 실행 계획 자체를 살펴봐야 한다. 모든 DBMS는 실행 계획을 조사하는 수단을 제공한다. MySQL의 경우에는 다음과 같다.  
  

```sql
EXPLAIN EXTENDED SQL 구문
```

  
명령어를 실행하면 기본적으로 3가지의 정보를 나타낸다.(이는 DBMS 마다 다를 수 있기 때문에 공식 문서를 참고해야 한다.)

-   **조작 대상 객체**: 현재 조작하고자 하는 객체로 테이블, 인덱스, 파티션, 시퀀스처럼 SQL 구문으로 조작할 수 있는 객체라면 무엇이라도 올 수 있다.
-   **객체에 대한 조작의 종류**: 객체에 대한 조작의 종류는 실행 계획에서 가장 중요한 부분이다.
-   **조작 대상이 되는 레코드 수**: 하나의 SQL 구문을 실행해도 여러 개의 조작이 수행된다. 이때 각 조작에서 처리가 필요한 레코드의 수를 의미한다. 여기서의 숫자는 카탈로그 매니저로 부터 얻은 값으로, 실제 SQL 구문을 실행한 시점의 테이블 레코드 수와 차이가 있을 수 있다. 즉, 옵티마이저는 실제 테이블을 보는 것이 아니라 통계라는 메타 정보만 바라보고 있다고 생각할 수 있다.

여기서 주의해야할 점은, 실행 계획에 출력되는 정보는 추정값이기 때문에 절대 지표로 사용하면 안 된 다는 것이다.

#### 테이블 결합의 실행 계획

SQL에서 지연이 일어나는 경우 중 상당 부분은 결합과 관련된 부분이다. 결합을 사용하면 실행 계획이 상당히 복잡해지므로, 옵티마이저도 최적의 실행 계획을 세우는 것이 어렵다.   
  
일반적으로 RDBMS는 결합을 할 때는 세 가지 종류의 알고리즘을 사용한다.   
  
가장 간단한 결합 알고리즘은 **Nested Loops**이다. 한쪽 테이블을 읽으면서 레코드 하나마다 결합 조건에 맞는 레코드를 다른 쪽 테이블에서 찾는 방식이다. 중첩 반복문으로 구현되므로 **Nested Loops**라는 이름이 붙었다.  
  
두 번째는 **Sort Merge**이다. 결합 키로 레코드를 정렬하고, 순차적으로 두 개의 테이블을 결합하는 방법이다. 결합 전에 전처리 단계에서 정렬을 수행해야 하는데, 이때 워킹 메모리(MySQL의 경우 정렬 버퍼)가 사용된다.  
  
세 번째는 **Hash**이다. 결합 키값을 해시로 맵핑하는 방법으로 해시 테이블이 필요하고 이를 위해 워킹 메모리를 사용한다.  
  
실제로 결합 SQL 구문을 실행하고 실행 계획을 살펴보면 위의 알고리즘 중 하나를 사용해서 결합을 수행하고 있는 것을 알 수 있다. 결합 시점의 실행 계획은 성능에 영향을 주기 때문에 다음 포스트에서 조금 더 자세히 다룰 것이다.

#### 정리

-   개발자는 데이터베이스에 SQL을 전달하는 것에서 끝난다.
-   데이터베이스는 옵티마이저를 통해서 최적의 실행 계획을 생성하게 되는데, 이때 올바른 카탈로그(통계 정보)가 성능에 중요한 영향을 미치게 된다.
-   통계 정보를 갱신하는 것은 매우 비용이 큰 작업이기 때문에 적절한 갱신 시점을 검토해야 한다.
-   실행 계획은 \`EXPLAIN\` 구문을 통해서 살펴볼 수 있으며 3 가지의 중요한 정보를 나타낸다.
-   SQL에서 지연이 일어나는 상당 부분은 결합과 관련된 부분이고, DBMS에서는 대표적은 세 가지 종류의 알고리즘을 사용한다.
