이전 글에서는 **브라우저에서 메시지를 송수신하는 과정**에 대해서 알아보았다. 

  
앞에서 배운 내용을 간단히 정리하면 애플리케이션(브라우저 등)은 데이터를 송수신하기 위해서는 OS의 프로토콜 스택에 HTTP 메시지를 위임하면서 데이터 송수신을 의뢰한다. 그리고 의뢰받은 OS의 프로토콜 스택은 Socket 라이브러리를 이용해서 소켓을 생성, 연결, 데이터 송·수신, 소켓 말소를 통해서 데이터를 송수신한다. 이렇게 송수신된 데이터는 애플리케이션에게 전달되게 된다. 

**그렇다면 데이터를 보낸다고 하는데, 실제로 케이블을 통해서 어떻게 디지털 데이터가 전달되는가?** 

이제 부터 **TCP의 관점**에서 조금 더 자세하게 알아보자.

애플리케이션이 데이터 송수신을 의뢰하는 OS의 **프로토콜 스택의 구조**는 다음과 같다.

![](/images/posts/tcp-ip-31/screenshot-2023-06-03-at-10-34-43-pm.png)

이전 글에서 데이터 송수신을 OS 프로토콜 스택에 의뢰하게 되면 소켓을 생성하고, 소켓에 통신에 필요한 정보를 저장한다고 했다.

**여기서 말하는 소켓은 물리적인 것인가, 논리적인 것인가? 논리적이라면 어떻게 정보를 저장할 수 있을까?**

  
프로토콜 스택은 내부에 제어 정보를 저장하기 위한 메모리 영역을 가지고 있다. 그리고 통신을 제어하기 위한 제어하기 위한 정보를 기록하게 되는데, 통신 상대의 IP 주소, PORT 번호, 통신 동작의 진행 상태 등을 저장하게 된다. 앞의 정보를 이전글에서 보았다.  
  
그렇다. **소켓은 논리적인 개념으로 제어 정보가 저장된 물리적인 메모리 영역으로 생각할 수 있다.** 실제 프로토콜 스택은 소켓의 정보를 참조해서 동작하는데, 이때 참조하는 것이 통신 제어 정보가 저장된 메모리 영역(소켓)이다.  
  
위의 제어 정보 중에 **통신 동작의 진행 상태**가 기록된다고 했다. 이는 실제로 데이터를 송수신하는 과정에서 완벽하게 데이터의 송수신이 이루어지지 않는 상황이 있기 때문이다. 데이터를 송신하면 상대측에서 응답을 보내게 되는데, 일정 시간이 경과해도 응답이 오지 않는 경우에는 데이터가 도착하지 못함을 판단하고 다시 데이터를 보내게 된다. 이때 송신 과정부터 어느 정도 시간이 경과했는지, 모른다면 무한히 응답을 기다리게 될 수 있기 때문에, 진행 상태를 소켓에 저장하게 된다.  
  
이제 논리적인 소켓에 대해서 조금 더 구체적으로 알았으니, 이전 글에서 간단히 알아본 **소켓을 실제로 생성하고 Socket 라이브러리의 connect 함수를 호출하는 부분부터 다시 살펴보자.**  
  
가장 먼저 소켓을 생성하는 부분이다. 애플리케이션에서 socket 함수를 통해서 소켓 생성을 프로토콜 스택에 의뢰하면, 프로토콜 스택은 **소켓 하나 크기의 메모리 영역을 확보**한다. 그리고 송수신을 시작하지 않은 초기 상태의 제어 정보를 확보한 메모리에 저장한다. 소켓이 만들어지면 확보한 메모리 영역(소켓)을 참조할 수 있는 디스크립터를 애플리케이션에게 반환한다. 디스크립터를 반환 받은 애플리케이션은 통신을 하기 위해서 프로토콜 스택에 소켓 생성을 계속 의뢰하지 않고, 디스크립터를 통해서 생성된 소켓을 참조하여 데이터 송수신을 의뢰한다.  
  
소켓이 생성된 후 애플리케이션은 **connect** 함수를 호출하고, 이때 프로토콜 스택은 접속을 시도한다.

접속은 통신 상대와의 사이에 제어 정보를 주고 받고, 소켓에 필요한 정보를 기록하고 데이터 송수신이 가능한 상태로 만드는 것이다. 

**소켓을 생성했으면 바로 데이터를 송수신 하면 되지, 접속을 꼭 해야 하는가?**

  
앞서 설명했듯이 데이터를 송신하면 받드시 응답을 받을 수 없는 경우도 있다. 상대측이 인터넷에 연결되지 않은 경우, 웹 서버가 다운되어 통신을 할 수 없는 경우, 상대측으로 도달하기 위해 거치는 단말이 다운된 경우 등 다양한 경우가 현실 세계에서는 다분하기 때문이다.

  
**그렇다면 어떻게 상대측과 데이터 송수신이 정상적으로 할 수 있다는 것을 보장할 수 있을까?**

  
가장 간단한 방법은 상대측에 물어보는 것이다. 예를 들어서 식당의 예약을 잡는다고 생각해 보자. 가장 먼저 식당에 전화를 할 것이다. 그리고 식당에서 전화를 받으면 언제, 몇 명의 인원이 방분할 것인데 예약이 가능한지 물어본다. 그리고 식당에서 예약을 할 수 있다고 응답하면 자신의 이름으로 예약을 하고, 예약한 날짜에 식당에 방문하게 된다.   
  
데이터의 송수신도 위와 같다. 가장 먼저 상대측에 데이터를 송신하고 싶다는 것을 알린다. 이때 어느 IP의(예약자) port번호(예약자의 전화번호)로 부터 예약한다고 알린다. 이때 상대측은 요청을 통해서 상대측의 소켓에 통신 제어 정보(송신 측의 IP, Port 번호 등)를 저장하고, 데이터 수신 가능 여부를 응답하게 된다. 가능하다는 응답이 돌아오면 송신 측은 해당 정보를 소켓에 저장하고, 비로소 통신을 하기 위한 준비가 끝나게 되고, 이러한 과정을 접속한다고 한다.

  
**접속을 하기 위해서 Port 번호와 IP 등의 제어 정보를 함께 알린다고 했는데, 데이터에 포함되어서 전달되는 것인가?**

  
제어 정보는 크게 두 가지로 나눌 수 있고, 클라이언트와 서버가 연락하기 위한 제어 정보(**TCP 헤더**)와 이더넷과 IP의 제어 정보(**IP헤더**, **MAC 헤더**)가 있다. 이러한 제어 정보는 데이터에 포함하지 않고, 데이터의 맨 앞에 별도로 추가한다. 이렇게 전달되는 단위를 패킷이라고 하며, 아래의 그림과 같이 표현할 수 있다.

![](/images/posts/tcp-ip-31/screenshot-2023-06-05-at-11-53-04-pm.png)

**TCP 헤더**에는 다음과 같은 정보가 저장된다. 

1.  송신처 포트 번호
2.  수신처 포트 번호
3.  시퀀스 번호
4.  ACK 번호
5.  컨트롤 비트
6.  윈도우

실제로 위의 정보보다 더 많은 정보가 기록된다.  
  
다음으로 접속하는 과정에서 대해서 자세히 알아보자.  
connect 함수는 다음과 같이 호출하여 프로토콜 스택에 정보를 넘겨준다.

> connect(descriptor, server IP address, server port number)

프로토콜 스택은 IP 주소, Port 번호  등의 제어 정보를 기록한 헤더를 만든다.  헤더에는 컨트롤 비트에 대한 정보도 기록되는데, 송신 전 SYN 비트를 1로 만들어서 기록한다.   
  
이렇게 만들어진 TCP 헤더와 데이터를 IP 담당 부분에 넘기고 송신을 의뢰한다. 그러면 IP 담당 부분이 이 패킷을 네트워크를 통해서 서버에 전달하고, 서버에 전달된 패킷은 서버 측 TCP에 전달된다. 전달된 패킷의 TCP 헤더에 기록된 수신처의 포트 번호를 통해서 서버 측의 접속 소켓을 찾고 필요한 정보를 소켓에 저장한다. 소켓에 정보를 모두 저장했다면, TCP 부분은 클라이언트 측으로 응답을 보낸다. TCP에서 응답을 주는 것은 클라이언트와 동일하지만, 여기서 ACK 비트를 1로 만들어서 TCP 헤더에 기록한다. 이는 서버 측이 정상적으로 패킷을 수신했다는 표시다.  
  
클라이언트는 응답받은 메시지를 IP 담당 부분을 거쳐 TCP 담당 부분으로 전달하고, TCP 헤더의 SYN와 ACK 비트를 검사하고 정상적으로 받은 응답의 정보를 소켓에 저장한다. 그리고 다시 서버 측에 응답을 잘 받았다는 것을 알리기 위한 메시지를 송신한다. 이때도 TCP 헤더의 ACK 비트를 1로 만들어 기록한다.  
  
이렇게 하면 클라이언트의 소켓과 서버의 소켓이 서로 데이터를 송수신할 수 있는 상태가 되고, 이를 파이프로 연결되었다고 표현한다. 위의 과정을 다르게 표현하면 **3-way Handshake**라고 부른다.

  
**드디어 서로의 존재를 알고 연결되었다. 이제는 메시지를 보내자.**

  
접속이 완료되면, write 함수를 호출하여 데이터 송수신을 시작한다.  애플리케이션은 데이터를 프로토콜 스택으로 전달하고 프로토콜 스택은 내부의 송신용 버퍼에 데이터를 일시적으로 저장한다. (데이터를 받은 즉시 송신하게 되면 작은 패킷을 많이 보내게 되고, 네트워크 이용 효율이 떨어진다)   
  
프로토콜 스택은 2가지의 기준으로 적절한 패킷을 생성 후 데이터를 송신하게 된다.   
  
첫 번째로 **MTU(Maximum Transmission Unit) 매개** 변수를 기준으로 패킷 사이즈를 결정하고 데이터를 송신한다. **MTU**는 한 패킷으로 운반할 수 있는 디지털 데이터의 최대 길이로, 헤더와 데이터를 포함한 길이를 의미한다. 그리고 헤더를 제외한 데이터의 크기를 **MSS(Maximum Segment Size)**라고 한다. MSS에 최대로 데이터를 채워서 패킷을 생성하면, 네트워크 이용 효율 저하를 방지할 수 있다.   
  
두 번째로 **타이밍**을 기준으로 데이터를 송신한다. **MSS**를 채우기 위해서 지속해서 기다리게 되면, 송신 속도 저하가 발생하기 때문에 **프로토콜 스택 내부의 타이머로 일정 시간 이상 경과하면 패킷을 송신**한다.  
  
실제로 프로토콜 스택은 두 가지의 기준을 적절히 절충해서 패킷을 전송한다.  
  
만약 데이터의 크기가 MSS를 넘어가게 되면 적정한 크기로 데이터를 쪼개서 송신하게 된다. TCP 담당 부분에서는 데이터를 조각으로 만든 뒤, 조각이 몇 번째 바이트의 조각인지 **시퀀스 번호를 헤더의 시퀀스 항목**에 기록한다. 예를 들어서 첫 번째 조각은 1로 기록할 수 있다.

시퀀스 번호는 수신 측에서 중요한 정보다. 만약 네트워크의 상황으로 인해 데이터가 순서대로 도착하지 않는 경우, 데이터가 소실된 경우 등이 발생했다고 가정하자. 만약 수신 측이 현제 데이터가 정상적으로 도착하지 않았다는 것을 알지 못하면, 도착한 순서대로 데이터를 조합 후 애플리케이션에 반환할 것이다. 그러면 요청이 원하는 목적과 다른 결과를 만들 수 있다. 

**그렇다면 누락에 대한 검증을 할 수 없을까?**

수신 측은 **시퀀스 번호**를 통해서 데이터의 누락을 알 수 있다. 만약 시퀀스 번호가 정상적으로 도착하지 않았다면, 헤더의 ACK 번호에 엉뚱한 값을 입력해서 응답하게 된다. 이때 **ACK 번호**는 헤더를 제외하고 **지금까지 받은 데이터의 바이트 크기**를 의미한다. 이러한 응답을 받은 송신 측은 데이터가 누락되었다는 것을 알고, 누락된 데이터부터 다시 송신하게 된다. 반대로 정상적으로 데이터를 수신하게 되면, 정상적인  **ACK 번호**에 기록해서 송신 측에 응답하게 된다. 이러한 응답을 **수신 확인 응답**이라고 한다.  
  
실제로 **시퀀스 번호**는 **난수를 바탕으로 초기값을 설정**하게 된다. 하지만 서버 측에서는 난수로 생성된 값이 초기값인지 알 수 없다. 그렇기 때문에 접속시, SYN의 비트를 1로 만드는 것 이외에도 시퀀스 항목에 난수인 초기값을 기록해둔다. 그러면 서버측에서 헤더의 시퀀스의 값을 통해서 앞으로 올 데이터의 초기 시퀀스 값을 알 수 있다.   
  
위의 내용을 바탕으로 정리하면 다음과 같다.

1.  접속동작을 실행할 때, 클라이언트는 서버측에 SYN 비트를 1로 하고 시퀀스 초기값을 기록한 패킷을 송신한다.
2.  서버는 수신한 패킷을 통해서 ACK 번호를 생성하고, ACK 비트를 1로 한 패킷을 다시 클라이언트로 송신한다. 이때 서버 측의 시퀀스 초기값을 패킷에 기록한다.
3.  클라이언트는 수신한 패킷을 통해서 ACK 번호를 생성 후 패킷을 서버 측에 송신한다.

![](/images/posts/tcp-ip-31/screenshot-2023-06-03-at-10-49-44-pm.png)

접속이 끝나면 데이터의 송수신이 시작된다.

1.  클라이언트에서 데이터를 포함한 패킷이 서버측에 전달되면, 시퀀스 번호와 데이터의 크기를 통해서 ACK 번호를 생성한다.
2.  서버는 생성한 ACK 번호를 패킷에 기록한 후 클라이언트로 송신한다.
3.  데이터를 전부 송신할 때까지 반복한다.

![](/images/posts/tcp-ip-31/screenshot-2023-06-03-at-10-51-59-pm.png)

이러한 과정은 마치 핑퐁을 하는 것과 같이 데이터를 받았는지를 계속 검사하면서 데이터를 송수신하기 때문에 **회복처리**를 TCP 담당 부분에서 간단하게 해결할 수 있다. 만약 일정 시간 동안 ACK 번호가 돌아오지 않으면 패킷을 다시 보내게 되는데, 이를 위해서 TCP는 송신용 버퍼 메모리에 송신한 패킷을 임시로 저장해 둔다.   
  
위의 구조는 회복 처리에 강력하고 데이터의 신뢰성이 높다. 하지만 데이터를 송신하고 ACK 번호 응답을 기다리고, 응답이 오면 다시 데이터를 송신하는 과정에서 ACK 번호를 가만히 기다리기만 하면 **시간이 낭비**된다.  
  
이러한 낭비되는 시간을 줄이기 위해서 TCP는 **윈도우 제어 방식**을 사용한다.

  
**윈도우 제어 방식**은 패킷을 보내고 **ACK 번호를 기다리지 않고**, 다음 패킷을 계속해서 보내는 방식이다.

  
**그렇다면 모든 데이터를 연속해서 쉬지 않고 보내면 되지 않을까?**

  
**그렇다면 좋겠지만 실제로는 그렇지 못하다.** 수신 측에서 수신된 패킷을 **수신용 버퍼 메모리에 임시로 저장**하고, 이를 순차적으로 처리한다. 처리된 데이터에 해당하는 응답 메시지를 보내고 다음 처리를 이어가는데, 수신용 버퍼 메모리 용량을 넘어서는 패킷이 수신되게 되면 메모리에 담지 못하고 버리게 된다. 이러면 데이터가 수신되지 못한 것과 같이 오류로 판단되게 된다. 이를 방지하기 위해서 수신 측은 송신된 데이터를 처리하고 ACK 번호를 응답할 때 TCP 헤더의 **윈도우 필드**에 **수신 버퍼의 허용 용량**을 함께 기록해서 응답하게 된다. 클라이언트는 윈도우 필드의 값을 참조하여 패킷을 송신하게 되면서 양방향 소통이 되게 된다.  
  
클라이언트의 데이터 수신은 위의 과정과 유사하고, **read**함수를 호출하면 데이터의 수신을 프로토콜 스택에 의뢰한다. 만약 송신 직후에 함수를 호출하면 아직 데이터가 수신되지 않을 수 있기 때문에, 프로토콜 스택은 작업을 보류한다. 그리고 데이터가 수신되면 TCP에서 데이터의 누락에 대해서 검사 후 정상적으로 수신된 데이터에 한해서 ACK 번호를 응답한다. 이렇게 받은 수신 데이터를 헤더에서 추출 후 다시 조립하고, 애플리케이션이 지정한 메모리에 담은 뒤 제어권을 애플리케이션에 넘겨주게 된다.  
  
모든 데이터의 송수신이 끝나면 소켓을 말소한다. 소켓의 말소는 클라리언트와 서버 측 어느 곳에서든 먼저 일어날 수 있다. 만약 서버 측이 먼저 **close**함수를 통해 소켓을 말소하면, TCP 헤더에 **FIN 비트**를 1로 기록 후 클라이언트에 응답하게 된다. 클라이언트는 해당 메시지에 대한 ACK 번호를 생성 후 응답하고, 애플리케이션에 데이터 송수신 종료를 알린다. 애플리케이션은 **close**함수를 호출하여 소켓을 말소하고, 서버 측에 FIN 비트를 1로 기록한 응답을 보내고 ACK 번호를 응답받음으로써 서버와의 대화를 종료한다.  
  
여기까지가 브라우저의 메시지 의뢰를 받아 프로토콜 스택이 데이터 송수신의 전체 과정이다.

  
**하지만 가장 처음의 의문인 케이블을 통해서 실제 디지털 데이터가 전달되는 것에 대한 답을 찾기에는 부족하다.**
