웹 개발을 공부하다 보면 다음과 같은 그림을 많이 보게 된다. 

![](/images/posts/post-29/screenshot-2023-05-26-at-10-22-41-pm.png)

클라이언트는 서버에 원하는 정보를 요청하고, 서버는 요청에 대한 적절한 응답을 클라이언트로 보내주는 것을 나타낸 그림이다.

  
**그렇다면 인터넷에서 두 개의 컴퓨터는 어떻게 통신을 하게 되는 것일까?** 

가장 먼저 웹 브라우저가 동작하는 것에서부터 알아보자. 웹 브라우저는 크게 다음과 같이 동작하게 된다.

1.  HTTP 리퀘스트 메시지를 작성한다.
2.  웹 서버의 IP 주소를 DNS 서버에서 조회한다.
3.  프로토콜 스택에 리퀘스트 메시지 송신을 의뢰한다.

처음부터 하나하나 살펴보자.

## \[HTTP 리퀘스트 메시지\]

우리는 인터넷을 사용할 때 다양한 브라우저를 사용한다. 그리고 브라우저에 무엇인가 요청하기 위해서는 다음과 같이 URL을 입력하게 된다. 

![](/images/posts/post-29/screenshot-2023-05-26-at-11-43-49-pm.png)

  
여기서 **URL(Uniform Resource Locator)**은 네트워크 상에서의 자원의 위치를 알려주기 위한 규약이다. 다르게 말하면 웹 리소스(웹 페이지 등)에 대한 참조이다. 

기본적으로 URL은 데이터에 액세스 하는 방법(**scheme**)을 나타내는 **_http_**, **_ftp_**, **_file_**, **_mailto_** 등으로 시작하게 된다. 브라우저는 단순히 웹 서버에 액세스 하는 것을 넘어서 파일 다운로드/업로드, 메일 등 다양한 기능을 포함하고 있기 때문에, URL은 어떠한 기능을 사용해서 데이터에 액세스 하면 좋은지 명시해 준다.

-   http: 웹 서버에 액세스 하는 경우
-   ftp: 파일을 다운로드 / 업로드하는 경우
-   file: 클라이언트 PC 자체의 파일에서 데이터를 읽어오는 경우
-   mailto: 메일을 송신하는 경우

**URL의 기본 구조**는 다음과 같다.

![](/images/posts/post-29/img.png)

https://www.ietf.org/rfc/rfc3986.txt

다시 돌아와서 브라우저에 URL 입력하고 엔터를 치게 되면, 브라우저는 가장 먼저 URL을 해석한다. 웹 서버의 요청의 경우에는 URL을 해석해서 **도메인**과 **요청하는 파일의 경로**, **쿼리 스트링** **등**을 각각 분해한다. 해석이 끝나면 어떤 웹 서버에 액세스 해야 하는지가 정해지고 HTTP 프로토콜을 사용하여 웹 서버에 액세스 하게 된다.  
  
**HTTP 프로토콜**은 클라이언트와 웹 서버가 주고받는 메시지의 내용이나 순서를 정한 것이다.   
먼저, 클라이언트는 웹 서버에게 리퀘스트 메시지를 보내게 되고, **HTTP 메시지의 포맷은** 다음과 같다.

![](/images/posts/post-29/img-1.png)

https://datatracker.ietf.org/doc/html/rfc7230#section-3

**start-line** 은  **request-line**(요청 시)과 **status-line**(응답 시)으로 나뉘게 된다. 그래서 클라이언트에서 요청을 보낼 때 생성하는 리퀘스트 메시지의 포맷은 다음과 같다.

> method SP request-target SP HTTP-version CRLF  
> \*( header-field CRLF )  
> CRLF  
> \[ message-body \]

**Request-target**에 사용자가 원하는 ‘**무엇**’이, **method**에 ‘**어떻게**’가 들어가게 된다. **Header-field**에는 리퀘스트에 대한 정보를 나타내고, **message-body**에는 웹 서버로 보내는 폼 데이터, JSON 데이터 등을 담을 수 있다. 이렇게 생성된 리퀘스트 메시지를 웹 서버로 보내면, 웹 서버로 부터 응답 메시지가 돌아오게 된다.

웹 서버로 부터 받는 응답 메시지의 기본 구조도 위와 같다. 단, **start-line**이 **status-line**으로 작성된다. **status-line**은 **request-line**과는 다르게 리퀘스트의 실행 결과에 대한 정보를 담게 된다. 

> HTTP-version SP status-code SP reason-phrase CRLF

**status-code**에는 리퀘스트의 실행 결과에 대한 상태 코드가 담겨 있고, **reason-pharse**에는 상태 코드 및 실행 결과에 대한 짧은 문장이 담겨 있다. 

응답 메시지를 받게 되면, 브라우저는 메시지에서 필요한 데이터를 추출하여 화면에 표시하게 되고 결과적으로 웹 페이지가 화면에 그려지게 된다. (만약 응답 메시지로 받은 HTML에 영상과 이미지 같은 태그 제어 정보가 포함되어 있으면 브라우저는 각 태그에 대한 정보를 받기 위해서 리퀘스트 메시지 생성 -> 응답 메시지 해석을 반복하게 된다.)



이렇게 브라우저는 웹 서버와 대화를 하게 된다. 

**하지만 리퀘스트 메시지와 응답 메시지는 어떻게 브라우저와 웹 서버 사이를 오갈 수 있을까?**

## \[웹 서버의 IP 주소를 DNS 서버에서 조회\]

브라우저는 URL을 해석하고 HTTP 메시지를 만들 수 있지만, 실질적으로 네트워크 송출 기능은 없다. 그렇다면 브라우저가 생성한 HTTP 메시지는 어떻게 네트워크로 웹 서버에게 전달될 수 있을까? 브라우저는 HTTP 메시지를 만들어서 **OS(Operating system)**에게 메시지 송신을 위임한다. 메시지를 전달받은 OS는 메시지에 작성된 IP 주소를 기반으로 목적지인 웹 서버로 메시지를 송신하게 된다.

여기서 이상한 점이 있다. **HTTP 메시지의 포맷을 보면 IP 주소를 작성하는 곳은 없는데, 어떻게 OS에서 IP 주소를 기반으로 웹 서버를 찾아갈 수 있는 것인가?**

이러한 의문을 해결하기 위해서는, 먼저 IP 주소에 대해서 알아야 할 필요가 있다.

인터넷이나 LAN은 **TCP/IP**를 기반으로 만들어졌다. TCP/IP는 **서브넷**이라는 작은 네트워크를 **라우터**와 연결하여 전체 군집을 만들었다고 생각할 수 있다. 여기서의 서브넷은 허브에 여러 대의 컴퓨터가 연결된 하나의 묶음 단위라고 생각할 수 있다. 아래의 그림을 참고하자.

![](/images/posts/post-29/screenshot-2023-05-27-at-11-02-14-pm.png)

서브넷에는 아파트 단지로 비교하면 **'xx 동 xx 호'**의 형태의 네트워크 주소가 할당되게 된다. 이때, **xx 동**은 **네트워크 번호**, **xx 호**는 **호스트 번호**에 해당한다.

그리고 **네트워크 번호와 호스트 번호가 합쳐진 것이 IP 주소**이다. 실제 IP 주소는 32 비트의 데이터로, 8비트씩 점으로 구분하여 10진수로 표기해 준다.

> 예시: 192.168.0.0

정리하면 IP 주소는 메시지가 도달하고자 하는 목적지의 주소라고 생각할 수 있다.

하지만 아직 HTTP 메시지의 포맷 어디에도 IP 주소를 찾을 수 없다. 대신 **도메인 명**을 작성하는데, 실제로 **도메인 명이 IP 주소의 이름**이라고 생각할 수 있다. (Map의 key와 value에서 도메인 명이 key, IP 주소가 value에 해당할 수 있다)

**왜 도메인 명을 IP 주소 대신 작성할까?** 이는 우리가 일상생활에서 전화번호를 얼마나 외우고 있는지 생각해 보면 쉽다. 요즘에는 전화번호를 저장하고 전화번호의 이름에 해당하는 상대방의 이름을 선택해서 전화를 건다. IP 주소가 전화로 비유하면 전화번호에 해당하고, 도메인명은 상대방의 이름에 해당하게 된다. 만약 전화번호가 바뀌어도 11자리의 숫자를 기억할 필요 없이, 이름에 해당하는 전화번호를 변경하고 저장해 두기만 하면 언제든지 상대방과 통화를 할 수 있다. 이렇듯 IP 주소는 기억하기 어렵고, 변경 가능성이 있기 때문에 IP 주소에 도메인 명이라는 이름을 붙여 메시지에 담아주게 되는 것이다.

이렇게 도메인 명과 IP 주소를 대응시키는 시스템을 **DNS(Domain Name System)**라고 부른다. 그리고 브라우저는 도메인 명에 대응하는 IP 주소를 가까운 DNS 서버에서 찾아와 OS의 프로토콜 스택에 메시지 송신을 위임한다.

## \[프로토콜 스택에 리퀘스트 메시지 송신을 의뢰\]

OS의 **프로토콜 스택**은 **Socket** 라이브러리를 이용하여 데이터(HTTP 메시지도 디지털 데이터다)를 송수신하게 된다. 

데이터의 송·수신 과정은 다음과 같이 요약할 수 있다.

1.  소켓 생성 (소켓은 데이터 송·수신의 출입구로 생각할 수 있다)
2.  서버 측 소켓에 파이프 연결
3.  데이터 송·수신
4.  파이프 분리 후 소켓 말소

과정을 조금 더 자세히 따라가 보자.

클라이언트는 소켓을 라이브러리를 호출하여 소켓을 생성한다. 소켓을 생성하면 소켓을 식별하기 위한 **디스크립터**가 반환되는데, 애플리케이션(여기서는 브라우저)은 반환된 디스크립터를 메모리에 저장해 둔다. 실제 컴퓨터에서는 여러 애플리케이션이 수많은 소켓을 생성하고 말소하기 때문에 각 **소켓을 식별하기 위한 것**이 필요한데, 이것이 **디스크립터**다.

소켓을 생성하면 프로토콜 스택에 서버 측의 소켓에 접속할 수 있도록 의뢰를 하고, Socket 라이브러리의 **connect** 함수를 호출한다. 이때, connect 함수를 호출하면서 **디스크립터**, **서버의 IP 주소**, **포트 번호**를 넘기게 된다. 이때 넘겨준 디스크립터를 기반으로 어떤 소켓을 서버 측과 연결할 것인지 결정하고, IP 주소로 웹 서버에 접속한다.

여기서 **'실제 컴퓨터에는 여러 애플리케이션이 수많은 소켓을 생성하고 말소한다'**는 문장을 다시 생각해 보자. **웹 서버에 도달은 했는데, 웹 서버의 어느 소켓에 파이프를 연결해야 할지 어떻게 알 수 있을까?** 이때 **상대방 측의 소켓을 식별하기 위해서 사용되는 것**이 **포트 번호**다.

포트 번호는 애플리케이션에 따라서 미리 정해져 있다. 예로 웹은 80 포트를 사용하고 이메일은 25번 포트를 사용한다. 

이렇게 포트 번호를 이용해 접속해야 할 소켓을 식별할 수 있고, 클라이언트의 포트 번호는 프로토콜 스택에서 적당한 값을 생성해서 서버 측에 전달해 주고 각 소켓이 연결되게 된다. 연결이 되면 소켓에는 상대측의 IP 주소, 포트 번호등을 기록한다.

소켓이 연결되면 데이터를 송·수신하는 것은 간단하다. 두 소켓이 연결되면 socket 라이브러리의 **write** 함수를 호출하고, 프로토콜 스택이 데이터를 서버에게 송신한다. 서버는 데이터를 수신 후 적절한 처리 후 응답 메시지를 송신하고 **close** 함수를 통해서 연결을 끊는다.

클라이언트에서는 **read** 함수를 통해 메시지를 수신하고 **수신 버퍼**에 메시지를 저장 후 애플리케이션에 메시지를 건네주고 close 함수를 통해 연결을 끊게 된다. 아래의 그림을 통해서 클라이언트와 서버의 소켓 통신을 실행 흐름을 확인할 수 있다.
